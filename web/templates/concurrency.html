🔐 Concurrency Control (Locking Mechanisms)

How our Accident Alert System handles simultaneous database access using shared & exclusive locks

🔐 Why Concurrency Control?
Problem: In our Accident Alert System, multiple dispatchers can be logged in simultaneously. Without concurrency control, two dispatchers could assign the same ambulance to different accidents — causing data corruption.
Locking mechanisms control concurrent access to data, preventing conflicts when multiple users or processes access the database at the same time. MySQL's InnoDB engine provides two levels of locking:
📋
Table-Level Locking
Locks the entire table. Simple but blocks all other sessions. Used for bulk operations like fleet reset.

Row-Level Locking
Locks only specific rows. Other rows remain accessible. Used for ambulance dispatch (lock one ambulance row).

Lock Compatibility Matrix

Requesting ↓ / Held → Shared (S) Exclusive (X)
Shared (S) ✅ Compatible — multiple readers allowed ❌ Conflict — writer blocks readers
Exclusive (X) ❌ Conflict — reader blocks writers ❌ Conflict — only one writer at a time
📋 Table-Level Locking
Definition: Locks the entire table. No other session can access the locked table (for WRITE lock) or modify it (for READ lock). Used for bulk operations where row-level locking would be too granular.

Table-Level READ Lock (Shared)

Multiple sessions can read simultaneously, but nobody can write until lock is released.
LOCK TABLES ... READ — Shared Table Lock
-- Session A: Lock hospitals table for reading
LOCK TABLES hospitals READ;

-- Session A CAN read:
SELECT * FROM hospitals;  -- ✅ Allowed

-- Session A CANNOT write:
UPDATE hospitals SET available_beds = 10 WHERE hospital_id = 1;
-- ❌ ERROR: Table 'hospitals' was locked with a READ lock

-- Session B CAN also read (shared lock is compatible):
SELECT * FROM hospitals;  -- ✅ Allowed

-- Session B CANNOT write:
UPDATE hospitals SET available_beds = 5 WHERE hospital_id = 2;
-- ⏳ BLOCKED — waits until Session A releases lock

UNLOCK TABLES;  -- Release the lock → Session B's UPDATE now proceeds

Table-Level WRITE Lock (Exclusive)

Only the lock-holding session can read or write. All other sessions are blocked.
LOCK TABLES ... WRITE — Exclusive Table Lock
-- Session A: Lock ambulances table for writing
LOCK TABLES ambulances WRITE;

-- Session A CAN read and write:
SELECT * FROM ambulances;  -- ✅ Allowed
UPDATE ambulances SET status = 'green',
       assigned_accident_id = NULL,
       assigned_hospital_id = NULL;  -- ✅ Allowed (bulk fleet reset)

-- Session B CANNOT read OR write:
SELECT * FROM ambulances;
-- ⏳ BLOCKED — waits until WRITE lock is released

UNLOCK TABLES;  -- All blocked sessions now proceed
🚑
In Our Project — Bulk Fleet Reset:
When the admin clicks "Reset All Ambulances" on the Fleet page, we need to UPDATE all 12 ambulances to status='green' and clear all assignments. A TABLE-LEVEL WRITE LOCK ensures no dispatcher can read stale ambulance data during the reset.
🔑 Row-Level Locking
Definition: Locks only the specific rows being accessed. Other rows in the same table remain fully available. Provided by InnoDB engine. Allows much higher concurrency than table-level locks.

Row-Level Shared Lock — LOCK IN SHARE MODE

Allows this transaction to read the row. Other transactions CAN also read it (shared), but CANNOT update it until lock is released.
SELECT ... LOCK IN SHARE MODE — Check Hospital Beds
START TRANSACTION;

-- Lock Apollo Hospital row with a SHARED lock
-- Purpose: Read bed count to decide if this hospital can accept a patient
SELECT hospital_id, name, available_beds
FROM hospitals
WHERE hospital_id = 1
LOCK IN SHARE MODE;
-- Result: hospital_id=1, Apollo Hospital, available_beds=15

-- ✅ Other sessions CAN also SELECT ... LOCK IN SHARE MODE on this row
-- ❌ Other sessions CANNOT UPDATE this row until we COMMIT/ROLLBACK

-- Decision logic: if beds >= 1, route patient here
SELECT 'Apollo Hospital has beds — routing ambulance here' AS decision;

COMMIT; -- Shared lock released

Row-Level Exclusive Lock — FOR UPDATE

Locks the row exclusively. No other transaction can read (with FOR UPDATE) or write to this row. Critical for preventing double-dispatch.
SELECT ... FOR UPDATE — Dispatch Ambulance Safely
START TRANSACTION;

-- Lock ambulance #5 with an EXCLUSIVE lock before dispatching
SELECT ambulance_id, vehicle_number, status
FROM ambulances
WHERE ambulance_id = 5 AND status = 'green'
FOR UPDATE;  -- ← EXCLUSIVE lock on ambulance_id=5 ONLY

-- ✅ Other sessions CAN still read/update ambulance_id=1,2,3,4,6,7...
-- ❌ Other sessions CANNOT read (FOR UPDATE) or UPDATE ambulance_id=5
-- If another dispatcher tries to dispatch #5, they WAIT here

-- Safe to update — we have exclusive ownership:
UPDATE ambulances
SET status = 'red',
    assigned_accident_id = 5,
    assigned_hospital_id = 1
WHERE ambulance_id = 5;

COMMIT; -- Exclusive lock released, other sessions unblocked

Concurrent Dispatch — How Row Locking Prevents Double-Assignment

Session A (Dispatcher in Chennai HQ)
T1 START TRANSACTION;
T2 SELECT * FROM ambulances WHERE ambulance_id=5 AND status='green' FOR UPDATE; → Row locked ✓ (Ambulance #5 is available)
T4 UPDATE ambulances SET status='red', assigned_accident_id=10 WHERE ambulance_id=5;
T5 COMMIT; ✓ Ambulance #5 dispatched successfully
Session B (Dispatcher in Remote Office)
T1 START TRANSACTION;
T3 SELECT * FROM ambulances WHERE ambulance_id=5 AND status='green' FOR UPDATE; ⏳ BLOCKED — waiting for Session A's exclusive lock!
T6 → Lock released. Row re-read: status='red'. 0 rows returned.
T7 ROLLBACK; ✗ Ambulance #5 already dispatched. Pick another.
✅
Result: Row-level locking prevented both dispatchers from assigning ambulance #5. Session B was forced to wait, then saw the updated status and picked a different ambulance. No data corruption.

💀 Deadlock Detection & Prevention
Deadlock: Occurs when two transactions each hold a lock the other needs. Neither can proceed. MySQL's InnoDB automatically detects deadlocks and rolls back the "cheaper" transaction (the victim) to break the cycle.

Deadlock Example in Our System

Deadlock — Two Dispatchers Lock in Different Order
-- Session A (Dispatcher 1):               -- Session B (Dispatcher 2):
START TRANSACTION;                          START TRANSACTION;

-- A locks ambulance #1 first               -- B locks hospital #1 first
UPDATE ambulances SET status='red'         UPDATE hospitals SET available_beds=14
  WHERE ambulance_id=1;                       WHERE hospital_id=1;
-- ✅ Locks amb#1                           -- ✅ Locks hosp#1

-- A now needs hospital #1 (held by B)     -- B now needs ambulance #1 (held by A)
UPDATE hospitals SET available_beds=14     UPDATE ambulances SET status='red'
  WHERE hospital_id=1;                        WHERE ambulance_id=1;
-- ⏳ WAITS for B's lock on hosp#1         -- ⏳ WAITS for A's lock on amb#1

-- 💀 DEADLOCK DETECTED! MySQL rolls back one session.
-- ERROR 1213: Deadlock found when trying to get lock;
-- try restarting transaction

Prevention — Consistent Lock Ordering

Solution: Always Lock Tables in the SAME Order
-- Our project convention: accidents → ambulances → hospitals → audit_logs
-- ALL dispatchers follow this order. No circular waits → No deadlocks.

START TRANSACTION;

-- Step 1: Lock accident row first
SELECT * FROM accidents WHERE accident_id = 5 FOR UPDATE;

-- Step 2: Then lock ambulance row
SELECT * FROM ambulances WHERE ambulance_id = 3 FOR UPDATE;

-- Step 3: Then lock hospital row
SELECT * FROM hospitals WHERE hospital_id = 1 FOR UPDATE;

-- Now safely update all three in locked order:
UPDATE ambulances SET status='red', assigned_accident_id=5, assigned_hospital_id=1
  WHERE ambulance_id=3;
UPDATE hospitals SET available_beds = available_beds - 1 WHERE hospital_id=1;
INSERT INTO audit_logs (user_id, action, details)
  VALUES (1, 'DISPATCH', 'Amb #3 → Accident #5 → Apollo Hospital');

COMMIT;
📍 Where Locking Is Used In Our Project
🚑 Ambulance Dispatch — Row-Level Exclusive
X-LockRow-Level
When a dispatcher clicks "Dispatch", the Flask backend runs SELECT ... FOR UPDATE on the target ambulance row. This acquires an exclusive row lock, preventing any other dispatcher from dispatching the same ambulance simultaneously. The lock is held until conn.commit().
📁 web/app.py → /api/dispatch route
🏥 Hospital Bed Check — Row-Level Shared
S-LockRow-Level
Before routing a patient, the system reads hospitals.available_beds with LOCK IN SHARE MODE. Multiple dispatchers can check beds simultaneously (shared lock), but no one can UPDATE the bed count until the check completes, preventing overbooking.
📁 web/app.py → /api/hospitals route
🔄 Bulk Fleet Reset — Table-Level Write
X-LockTable-Level
init_ambulance_fleet() runs at server startup. It uses an implicit table-level write lock to reset all 12 ambulances to status='green' and assign valid Chennai coordinates. No dispatcher can access the fleet during this bulk operation.
📁 web/app.py → init_ambulance_fleet()
📊 Dashboard Stats — Table-Level Read
S-LockTable-Level
The dashboard polls /api/stats every 5 seconds. MySQL uses implicit shared locks during SELECT queries, allowing multiple dashboard sessions to read counts simultaneously without blocking each other.
📁 web/app.py → /api/stats route
🗄️ Connection Pool — Isolation Between Requests
Row-LevelImplicit
MySQLConnectionPool(pool_size=5, pool_reset_session=True) ensures each Flask request gets a clean connection. When a connection returns to the pool, any held locks are automatically released, preventing lock leaks.
📁 web/app.py → DB_CONFIG
🔒 Deadlock Prevention — Lock Ordering
X-LockRow-Level
All dispatch transactions follow the same lock order: accidents → ambulances → hospitals → audit_logs. This consistent ordering eliminates circular waits and prevents deadlocks across concurrent dispatch requests.
📁 web/app.py → all dispatch-related routes
