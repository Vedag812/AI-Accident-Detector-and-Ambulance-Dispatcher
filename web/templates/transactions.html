⚡ ACID Properties — Overview
In our Accident Alert System, transactions are critical. When an accident is detected and an ambulance is dispatched, multiple database operations must succeed or fail together. ACID properties ensure this reliability.
A
Atomicity
"All or Nothing"
Dispatching an ambulance updates 3 tables. If any one fails, ALL roll back.
C
Consistency
"Valid State to Valid State"
CHECK constraint ensures beds ≥ 0. No transaction can violate this.
I
Isolation
"No Interference"
Two dispatchers assigning the same ambulance won't corrupt data.
D
Durability
"Permanent Once Committed"
Once an accident is recorded and committed, it survives a server crash.
🚨
Real Scenario — Ambulance Dispatch Transaction:
① Report Accident → ② Assign Ambulance → ③ Create Request → ④ Log Audit → ✓ COMMIT
If step ③ fails → steps ①, ② are ROLLED BACK. No partial state.
A Atomicity — "All or Nothing"
Definition: A transaction is an indivisible unit. Either ALL operations succeed (COMMIT), or ALL are undone (ROLLBACK). There is no partial execution.

✅ Successful Transaction (COMMIT)

When AI detects an accident, the system dispatches an ambulance. This involves updating 3 tables atomically:
Ambulance Dispatch — Complete Transaction
START TRANSACTION;

-- Step 1: Insert the accident report
INSERT INTO accidents (location, latitude, longitude, vehicle_id, severity, description)
VALUES ('T Nagar Junction', 13.0418, 80.2341, 'TN22KL7841', 'Critical',
        'Side impact collision detected by YOLO');

-- Step 2: Assign the nearest available ambulance
UPDATE ambulances
SET status = 'red',
    assigned_accident_id = LAST_INSERT_ID(),
    assigned_hospital_id = 1
WHERE ambulance_id = 3 AND status = 'green';

-- Step 3: Create dispatch request record
INSERT INTO ambulance_requests (accident_id, ambulance_id, hospital_id, status, estimated_time)
VALUES (LAST_INSERT_ID(), 3, 1, 'Dispatched', 8);

-- Step 4: Log the action in audit trail
INSERT INTO audit_logs (user_id, action, details)
VALUES (1, 'AMBULANCE_DISPATCHED', 'Amb #3 → T Nagar Junction (Critical)');

COMMIT; -- ✅ All 4 operations succeed → changes are permanent

❌ Failed Transaction (ROLLBACK)

If any step fails (e.g., ambulance already assigned), the ENTIRE transaction is rolled back:
Ambulance Dispatch — Failure & Rollback
START TRANSACTION;

-- Step 1: Insert accident ✅ (succeeds)
INSERT INTO accidents (location, latitude, longitude, vehicle_id, severity)
VALUES ('Guindy Flyover', 13.0067, 80.2206, 'TN04XJ4598', 'Critical');

-- Step 2: Assign ambulance ❌ (FAILS — ambulance #3 is already status='red')
UPDATE ambulances
SET status = 'red', assigned_accident_id = LAST_INSERT_ID()
WHERE ambulance_id = 3 AND status = 'green';
-- 0 rows affected → ambulance was not available!

ROLLBACK; -- ❌ Step 1's INSERT is also undone. Database unchanged.

C Consistency — "Valid State to Valid State"
Definition: A transaction must take the database from one valid state to another valid state. All integrity constraints (PK, FK, CHECK, NOT NULL, UNIQUE) must hold before and after the transaction.

Constraints That Enforce Consistency

CHECK
available_beds >= 0 AND available_beds <= capacity
Prevents beds going negative when patients are admitted
FOREIGN KEY
ambulances.assigned_accident_id → accidents(accident_id)
Cannot assign ambulance to a non-existent accident
NOT NULL
accidents.location NOT NULL
Cannot report an accident without a location
UNIQUE
ambulances.vehicle_number UNIQUE
No two ambulances can share the same vehicle number

Example: CHECK Constraint Preventing Inconsistent State

Consistency Violation — Rejected by CHECK
START TRANSACTION;

-- Hospital has 15 available beds. Try to admit 20 patients:
UPDATE hospitals
SET available_beds = available_beds - 20
WHERE hospital_id = 1;

-- ❌ ERROR 3819: CHECK constraint 'hospitals_chk_1' is violated
-- available_beds would become -5, violating: available_beds >= 0
-- Transaction is rejected. Database stays in a VALID state.

ROLLBACK;
Consistency Maintained — FK Constraint
START TRANSACTION;

-- Try to assign ambulance to non-existent accident #9999:
UPDATE ambulances
SET assigned_accident_id = 9999
WHERE ambulance_id = 1;

-- ❌ ERROR 1452: Cannot add or update a child row:
-- a foreign key constraint fails (assigned_accident_id → accidents.accident_id)
-- Database remains CONSISTENT.

ROLLBACK;
✅
Consistency guarantees: No matter what operations a transaction performs, the database will never end up in a state that violates any constraint. If it would, the entire transaction is rejected.
I Isolation — "No Interference Between Transactions"
Definition: Concurrent transactions execute as if they were running serially (one after another). One transaction's intermediate state is invisible to other transactions.

Problem: Two Dispatchers Assign the Same Ambulance

Without isolation, two control room operators could dispatch the same ambulance simultaneously:
Session A (Dispatcher 1)
T1 START TRANSACTION;
T2 SELECT * FROM ambulances WHERE ambulance_id=3 AND status='green'; → Found! Available ✓
T5 UPDATE ambulances SET status='red', assigned_accident_id=10 WHERE ambulance_id=3;
T6 COMMIT; ✓ Success
Session B (Dispatcher 2)
T1 START TRANSACTION;
T3 SELECT * FROM ambulances WHERE ambulance_id=3 AND status='green'; → Found! (reads old state)
T4 UPDATE ambulances SET status='red', assigned_accident_id=11 WHERE ambulance_id=3; ⏳ BLOCKED (waiting for Session A's lock)
T7 → UPDATE changes 0 rows (status is already 'red')

Solution: SELECT ... FOR UPDATE

Isolation via Row-Level Locking
START TRANSACTION;

-- Lock the ambulance row so no other transaction can modify it
SELECT * FROM ambulances
WHERE ambulance_id = 3 AND status = 'green'
FOR UPDATE;  -- ← Acquires EXCLUSIVE lock on this row

-- Now safe to update — no other session can read or write this row
UPDATE ambulances
SET status = 'red', assigned_accident_id = 10
WHERE ambulance_id = 3;

COMMIT; -- Lock released

MySQL Isolation Levels

READ UNCOMMITTED
Can read uncommitted data (dirty reads)
READ COMMITTED
Only reads committed data
REPEATABLE READ MySQL Default
Same query returns same results within a transaction
SERIALIZABLE
Full serial execution — highest isolation, lowest concurrency
Check & Set Isolation Level
-- Check current isolation level
SELECT @@transaction_isolation;
-- Result: REPEATABLE-READ (MySQL default)

-- Set isolation level for current session
SET SESSION TRANSACTION ISOLATION LEVEL SERIALIZABLE;
D Durability — "Permanent Once Committed"
Definition: Once a transaction is committed, the changes are permanent and survive any subsequent system failure — power outage, crash, or hardware failure.

How MySQL Ensures Durability

📝
Write-Ahead Log (WAL)
InnoDB writes changes to the redo log BEFORE modifying data pages. If a crash occurs, the redo log replays committed transactions.
💾
innodb_flush_log_at_trx_commit
Set to 1 (default): Redo log is flushed to disk on EVERY commit. Maximum durability.
🔄
Doublewrite Buffer
InnoDB writes data pages to a doublewrite buffer before the actual tablespace, preventing partial page writes on crash.
🔁
Crash Recovery
On restart after crash, InnoDB automatically replays committed transactions from the redo log and rolls back uncommitted ones.
Durability in Action — Accident Report
START TRANSACTION;

INSERT INTO accidents (location, latitude, longitude, vehicle_id, severity)
VALUES ('Marina Beach', 13.0500, 80.2824, 'TN09BZ1100', 'High');

COMMIT;
-- ✅ This accident record is now PERMANENTLY stored.
-- Even if the server crashes 1 millisecond after COMMIT,
-- the record will be recovered from the redo log on restart.

-- Verify durability setting:
SHOW VARIABLES LIKE 'innodb_flush_log_at_trx_commit';
-- Value = 1 → Full durability (flush on every commit)
✅
Durability guarantee: Once the system confirms "Accident Reported" and the transaction is committed, that record is safe — permanently written to disk via InnoDB's WAL mechanism.
📍 Where ACID Is Used In Our Project
🚑 Ambulance Dispatch (app.py)
AtomicityConsistency
When the "Dispatch" button is clicked, Flask opens a DB connection, INSERTs into ambulance_requests, UPDATEs ambulances.status to 'red', and UPDATEs ambulances.assigned_accident_id. Uses conn.commit() only after ALL succeed. On any exception → conn.rollback().
📁 web/app.py → /api/dispatch route
🔍 YOLO Accident Detection (yolo.html)
AtomicityDurability
When YOLO detects a collision, it calls /api/report-accident which INSERTs into accidents table. The after_accident_audit trigger auto-inserts into audit_logs. Both happen atomically inside one transaction. Once committed, the accident record is durable.
📁 web/app.py → /api/report-accident route
🏥 Hospital Bed Update (Trigger)
ConsistencyIsolation
The update_bed_count trigger fires AFTER UPDATE on ambulances. When ambulance status changes from 'red' to 'green' (patient delivered), it auto-decrements hospitals.available_beds. The CHECK constraint available_beds >= 0 ensures consistency.
📁 SQL → update_bed_count trigger
👤 User Login (Session Management)
IsolationDurability
Login verifies credentials via SELECT, then auto-migrates plaintext passwords to SHA-256 hash via UPDATE — both in the same transaction. REPEATABLE READ isolation ensures no concurrent login can interfere.
📁 web/app.py → /login route
🗃️ Connection Pool (Pool Reset)
AtomicityIsolation
Flask uses MySQLConnectionPool(pool_size=5, pool_reset_session=True). Each connection is reset when returned to pool, ensuring no leftover transaction state leaks between requests.
📁 web/app.py → DB_CONFIG
📊 Fleet Initialization (Startup)
AtomicityConsistencyDurability
init_ambulance_fleet() runs at startup. It ensures exactly 12 ambulances exist with valid Chennai coordinates. All INSERT/UPDATE operations are committed as one transaction. If the server restarts, the fleet is durable.
📁 web/app.py → init_ambulance_fleet()
